# Fix: PostgreSQL Fresh Migration 被过长 Alembic Revision 阻断

## 0. 改的代码大致位置

- `alembic/versions/0008_collection_evidence_foundations.py`
- `tests/test_schema_migrations.py`

## 1. 为什么改

Alembic 默认创建的 `alembic_version.version_num` 是 `VARCHAR(32)`，但
`0008_collection_evidence_foundations` 长 36 个字符。PostgreSQL 在 0008 migration
本身执行完成后写入新 revision 时会严格检查长度，因此从空数据库执行
`alembic upgrade head` 会停在 0007 -> 0008，后续所有表和索引都无法创建。

现有 SQLite migration round-trip 没有暴露问题，因为 SQLite 不强制执行声明的
`VARCHAR(32)` 长度。问题由隔离 PostgreSQL schema 中的 C4 双会话并发测试首次稳定
复现，错误为 `StringDataRightTruncationError`。

这属于部署阻断级 P0：Docker、Linux native 和 Windows native 虽然共用同一套迁移，
但只要目标是全新 PostgreSQL schema，就无法启动到当前 head。

## 2. 改完预期效果如何

- 从空 PostgreSQL schema 执行完整迁移时，0008 可以正常登记长 revision。
- 0010 等超过 32 字符的后续 revision 也能继续写入。
- 已迁移或人工扩列的 PostgreSQL schema 重跑升级时保持兼容。
- SQLite 行为不变，标准 migration upgrade/downgrade/re-upgrade 仍可执行。
- downgrade 不把列缩回 32；执行 downgrade migration 时版本表中仍可能保存长
  revision，提前缩列会再次触发同类错误。

## 3. 如何改的/改进思路

先增加回归断言，确认首个超过 32 字符的 revision 必须在自身业务 DDL 之前扩展
Alembic 版本列。旧迁移缺少该操作，测试按预期失败。

随后在 0008 `upgrade()` 开头仅针对 PostgreSQL 执行
`ALTER COLUMN version_num TYPE VARCHAR(128)`。不能只新增一个 0012 修复迁移，因为全新
数据库在写入 0008 revision 时已经失败，永远无法到达链尾修复。修改后同时验证了：

- 完整 SQLite migration 测试；
- 本地 PostgreSQL 随机隔离 schema 从空库升级到 head；
- 隔离 schema 内两个独立 session 竞争同一 C4 scan slice。

测试结束后隔离 schema 使用 `DROP SCHEMA ... CASCADE` 清理，没有修改配置中的业务
schema。
