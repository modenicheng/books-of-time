# Fix: 幂等任务并发入队使调用方事务失败

## 0. 改的代码大致位置

- `books_of_time/db/repositories.py`
- `tests/test_task_queue.py`

## 1. 为什么改

`CollectionTaskRepository.enqueue()` 原先先查询相同 `idempotency_key` 的活跃任务，未
找到时直接插入。两个 scheduler 或 worker 事务并发入队同一 key 时，双方都可能在
对方提交前查到空结果，随后由数据库的 active partial unique index 拒绝其中一次插入。

唯一索引保证了不会真正留下重复任务，但原实现没有在 savepoint 中处理该
`IntegrityError`。失败事务因此无法继续物化 cohort 的其他 component，也无法由调用方
正常提交或回滚当前业务单元。在多 scheduler 的 PostgreSQL 长期服务中，这会把正常的
幂等竞争升级为整次计划失败。

## 2. 改完预期效果如何

- 并发事务仍只会保留一条相同 key 的活跃任务。
- 竞争失败方回滚局部插入后，重新读取并返回竞争赢家创建的任务。
- 调用方外层事务保持可用，可继续物化其余 cohort/component 证据。
- 如果冲突后不存在相同 key 的活跃任务，原始 `IntegrityError` 仍会抛出，不掩盖其他
  数据约束错误。
- 已存在任务与请求中的 cohort/component 归属不一致时，继续拒绝错误复用。
- 不带 `idempotency_key` 的普通任务保持原插入路径。

## 3. 如何改的/改进思路

先用确定性的竞态 session 测试模拟数据库行为：第一次查询返回空，flush 抛出唯一键
`IntegrityError`，第二次查询返回另一个事务已创建的活跃任务。原实现直接抛错，测试
按预期失败。

修复只对带幂等键的插入开启 `begin_nested()` savepoint。唯一约束竞争发生后，局部
回滚并使用非 `skip_locked` 的锁定查询加载赢家，再执行与普通复用路径相同的 cohort
归属校验。这样以数据库唯一索引作为最终仲裁，同时保留 repository 的调用方事务边界。
