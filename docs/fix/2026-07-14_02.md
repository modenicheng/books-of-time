# Fix: 失败响应 raw 存储异常被标为 abandoned

## 0. 改的代码大致位置

- `books_of_time/db/http_evidence.py`
- `tests/test_http_request_attempts.py`

## 1. 为什么改

HTTP client 收到 429、captcha 或 5xx response 后会先记录状态和时间，再由 evidence
sink 保存失败 body。原实现若 `raw_store.save()` 因磁盘、权限或对象存储故障抛错，
attempt 会保持 `started`，随后 worker 将其收口为 `abandoned`。这会把“已收到
response，但证据存储失败”误写成 collector 中断，后续按状态统计时无法准确识别
raw persistence 故障。

## 2. 改完预期效果如何

- 失败 response 的 raw 保存成功时仍为 `failed`，并关联 `raw_payload_id`。
- raw store 保存失败时 attempt 变为 `failed`、`error_type=raw_storage`，且 raw ID
  保持 NULL。
- `http_status`、响应时间等已经获得的 HTTP 证据继续保留。
- 原始存储异常继续向 worker 上抛，不被 RequestFailure 或成功状态掩盖。

## 3. 如何改的/改进思路

仅包裹同步 `raw_store.save()` 边界；捕获普通异常后，使用既有 attempt repository
写入 `failed/raw_storage` 和不含 URL/凭据的异常类型摘要，再原样 re-raise。worker
后续的 abandoned 收口只处理 `started`，因此不会覆盖该终态。回归测试使用故障
raw store 验证没有创建伪 raw 行，同时 attempt 保留 HTTP 429 并准确标记存储失败。
