# Fix: Windows 服务无法通过定向控制事件优雅停止

## 0. 改的代码大致位置

- `books_of_time/cli.py`
- `tests/test_cli.py`

## 1. 为什么改

Windows 长期服务原先只注册 `SIGINT` 和 `SIGTERM`。真实新进程组烟测中，向目标进程组发送 `CTRL_C_EVENT` 后，服务 20 秒内没有退出，`service_instances.status` 仍为 `running`。

Windows 使用 `CREATE_NEW_PROCESS_GROUP` 启动独立服务进程时，可定向发送 `CTRL_BREAK_EVENT`；Python 将它映射为 `SIGBREAK`。原实现没有注册 `SIGBREAK`，因此无法把该控制事件转成 `ServiceHost.request_stop()`。

## 2. 改完预期效果如何

- Windows 独立进程组可通过 `CTRL_BREAK_EVENT` 请求协作式停止。
- worker、scheduler 和 heartbeat 继续走既有优雅停止流程。
- 服务进程退出码为 0，数据库实例状态最终写为 `stopped`。
- Linux/macOS 的 `SIGINT`、`SIGTERM` 行为保持不变。

## 3. 如何改的/改进思路

新增 `_service_stop_signals()`，始终返回 `SIGINT`、`SIGTERM`，并在当前平台存在时加入 `SIGBREAK`。信号安装与清理继续使用同一套 loop/fallback 逻辑。单元测试验证 Windows 候选集合，随后用隔离 SQLite、Windows 新进程组和真实 `CTRL_BREAK_EVENT` 完成进程级烟测。
