# Fix: init-db 创建无 Alembic revision 数据库

## 0. 改的代码大致位置

- `books_of_time/db/schema.py`
- `books_of_time/cli.py`
- `alembic/env.py`
- `docs/DEPLOYMENT.md`
- `tests/test_schema_migrations.py`
- `tests/test_cli.py`

## 1. 为什么改

`bot init-db` 原先直接调用 `Base.metadata.create_all`。它会创建当时 ORM 中的表，但不会创建或更新 `alembic_version`。长期服务引入 schema revision 健康检查后，这类数据库会被 `service doctor` 判定为不可用；直接执行 `alembic upgrade head` 又会从 0001 重新建表和 PostgreSQL enum，与已有对象冲突。

真实开发库复现了该问题：数据表已存在，但没有 `alembic_version`；首次 upgrade 在创建 `taskkind` 时因对象已存在而失败。进一步审计还发现旧库缺少事件表、`frontier_states.extra` 及后续 enum 值，因此不能盲目 stamp。

## 2. 改完预期效果如何

- 新库执行 `bot init-db` 后直接处于 Alembic head，不再生成无版本数据库。
- 旧开发库只能通过显式 `bot init-db --adopt-legacy` 接管。
- 接管前严格比对 metadata；只允许已知新表、`frontier_states.extra` 和已知 enum 漂移，其他差异一律在写入前拒绝。
- 新建、Docker、Linux 原生和 Windows 开发共用同一迁移入口。
- 应用内调用 Alembic 不会重置已存在的 logging handlers。

## 3. 如何改的/改进思路

`create_schema()` 改为在独立线程中运行 Alembic `upgrade head`，避免与 CLI 已有 asyncio loop 冲突。Alembic env 可接收 programmatic 配置路径，并在应用内调用时跳过 `fileConfig`。

`adopt_legacy_schema()` 使用 Alembic `compare_metadata` 建立差异白名单，对已知 PostgreSQL enum 使用 autocommit `ADD VALUE IF NOT EXISTS`，补齐 `frontier_states.extra`后 stamp `0001_initial`，再正常执行后续 revision。审计引擎与 Alembic 同样遵守 `BOT_DATABASE_SCHEMA`，保证隔离 schema 验收的一致性。

验收包括：SQLite 新库和 legacy 往返、PostgreSQL 随机 schema legacy 接管、真实开发库升级到 `0004_comment_analysis_flags`、`service doctor` 全项通过，以及全量 `270 passed`。
