# Fix: Latest Scan 同切片重试成功后沿用旧 Frontier 状态

## 0. 改的代码大致位置

- `books_of_time/collectors/latest_scan.py`
- `tests/test_latest_scan_worker.py`

## 1. 为什么改

`LatestScanCollector._fetch_with_retry()` 在请求失败时会立即通过 CAS 持久化当前
cursor、失败次数和错误原因。CAS 会推进 `frontier_states.version`，并返回对应的新
`FrontierState` 和进度状态。

旧实现只在 `_fetch_with_retry()` 的局部变量中保存这些新状态。如果同一切片内的后续
请求成功，该方法只返回 `FetchResult`，外层 baseline tail、baseline head sweep 和
incremental 循环仍可能把调用前的旧 frontier/progress 传给页面持久化。代码因此隐式
依赖 SQLAlchemy identity map 恰好原地刷新同一个 ORM 实例；一旦 repository 合法返回
另一个实例，下一次 CAS 就会使用旧版本并产生不必要的 `FrontierVersionConflict`。

这会把一次已经成功的网络重试变成采集任务失败，并使本应写入的 raw page、评论和
media 证据无法完成持久化。问题影响三个 scan-backed latest 模式，属于采集可靠性
P1 缺陷。

## 2. 改完预期效果如何

- 同一切片内第一次请求失败、后续请求成功时，页面继续正常持久化。
- 成功后的 CAS 使用失败记录产生的最新 frontier version。
- 最新 progress 与 HTTP 结果一起交还调用者，不再依赖 ORM identity map 对象别名。
- 跨切片重试次数、重试耗尽 corrupted 终态和 legacy latest 路径保持不变。
- baseline tail、baseline head sweep 和 incremental 三种模式使用相同状态传递规则。

## 3. 如何改的/改进思路

先增加端到端回归测试：构造同一 cursor 第一次请求失败、第二次成功，并在第一次 CAS
前把旧 frontier 从 session identity map 中移除，模拟 repository 返回新的合法 ORM
实例。旧实现稳定在页面持久化阶段以旧版本触发 `FrontierVersionConflict`。

修复后，`_fetch_with_retry()` 成功返回三项状态：

```text
(FetchResult, latest FrontierState, latest progress)
```

三个扫描循环在调用 `_persist_page()` 前统一解包并替换本地状态。失败、暂停和重试耗尽
仍返回 `None`，沿用原有终态处理。修复后新回归测试通过，并重新执行 latest scan、
legacy latest、评论 repository 和 cohort materialization 相关测试。
