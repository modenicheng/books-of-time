# Fix: Cohort Skew 错用 Worker 任务开始时间

## 0. 改的代码大致位置

- `books_of_time/db/cohort_repositories.py`
- `books_of_time/db/http_evidence.py`
- `tests/test_worker_cohort_lifecycle.py`

## 1. 为什么改

`SnapshotCohortExecutionRepository.mark_task_started()` 原先在 worker 领取 task 时立即计算
`skew_seconds = task_started_at - scheduled_for`。但 task 已被领取不代表平台请求已经发出：
collector 还可能等待限流、读取状态、准备 cursor，甚至在发出任何 HTTP 请求前失败。

设计基线要求 checkpoint skew 使用首个真实 HTTP attempt 的 `request_started_at`。旧口径会
系统性低估请求延迟，还会为零请求的失败或测试任务伪造 skew，使 C7 容量判断和 C9
`checkpoint_max_skew_seconds` 验收建立在错误时间上。

## 2. 改完预期效果如何

- worker lease 仍立即把 component/cohort 标记为 running，并保留 task 生命周期的
  `started_at`。
- `skew_seconds` 在首个带 cohort/component 关联的 HTTP attempt 开始时写入。
- 后续分页、重试或第二个请求不会覆盖首个请求的 skew。
- task 在没有发出 HTTP 请求的情况下结束时，`skew_seconds` 保持 NULL，明确表示没有
  可用请求开始证据。
- 旧任务或独立 HTTP attempt 的 cohort/component ID 均为 NULL，不触发额外查询或写入。

## 3. 如何改的/改进思路

先修改 worker 生命周期回归：无 HTTP 的成功 collector 必须保留 NULL skew；另一个
collector 在 task 开始后第 7 秒和第 12 秒登记两次 attempt，预期 skew 使用第一请求，
即相对提前 10 秒的 `scheduled_for` 为 17 秒。旧实现分别得到 10 秒和 10 秒，两个测试
均按预期失败。

修复保留 `mark_task_started()` 的状态转换，但移除其中的 skew 计算。数据库 HTTP
evidence sink 在 attempt 行成功插入后调用 execution repository；repository 锁定并校验
cohort/component，只在 `skew_seconds IS NULL` 时用该 attempt 的 `request_started_at`
计算一次。attempt、component 和 task 仍处于同一 worker 数据库事务中。
