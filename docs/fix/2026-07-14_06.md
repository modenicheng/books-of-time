# Fix: Planner 候选批次导致未接管视频饥饿

## 0. 改的代码大致位置

- `books_of_time/db/cohort_repositories.py`
- `tests/test_snapshot_cohort_planner.py`

## 1. 为什么改

`VideoCollectionStateRepository.list_candidates()` 原先始终按
`KnownVideo.first_seen_at` 从旧到新截取固定批次。只要已知视频数量超过
`batch_limit`，最老的一批视频就会在每次 30 秒规划中反复占满候选集，排在批次之后的
新视频永远没有机会建立 `video_collection_states`，也不会生成 routine、checkpoint 或
recovery cohort。

默认批次上限为 5000，因此该问题会在长期服务积累足够视频后造成静默采集缺口。它不
影响 planner 进程存活，也不会产生明显异常，但会让批次之外的视频永久失去 C3 规划
证据，属于采集完整性问题。

## 2. 改完预期效果如何

- 尚未建立 collection state 的视频优先进入候选批次。
- 某个视频首次被接管后，会让位给仍未接管的视频，不再永久占用有限批次。
- 全部视频完成接管后，已接管视频按最早 `next_due_at` 优先规划。
- `next_due_at` 尚未建立的异常/过渡状态排在已有明确到期时间的状态之后。
- 同一优先级内仍按 `first_seen_at` 和 `bvid` 保持稳定、可复现排序。

## 3. 如何改的/改进思路

先增加 `batch_limit=1` 的两轮 planner 回归测试：第一轮接管最老视频，第二轮必须接管
仍未见过的视频。原实现第二轮继续返回最老视频，测试按预期失败。

修复在 `known_videos` 与 `video_collection_states` 之间使用 outer join，并增加两级稳定
排序：未接管行优先；已接管行再按非空且最早的 `next_due_at` 优先。该改动不扩大单次
批次、不改变 planner 事务边界，也不改变 cohort 的幂等键。
