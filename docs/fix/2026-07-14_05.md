# Fix: Latest Follow-up 丢失 Cohort 关联

## 0. 改的代码大致位置

- `books_of_time/collectors/latest_comments.py`
- `tests/test_latest_comments_worker.py`

## 1. 为什么改

latest collector 在时间预算用尽、页面重试暂停或 baseline/head/incremental 扫描需要继续
时，会通过 `_enqueue_followup()` 创建下一条 task。C3 首任务已经携带
`snapshot_cohort_id` 和 `snapshot_cohort_component_id`，但原 follow-up 只复制 kind、
target、priority、payload、budget 和 retry 配置，没有复制这两个关联字段。

结果是 follow-up 的 coverage 和 HTTP attempt 重新变为 NULL，worker 也把它当成旧式
独立任务，不再推进原 component。长扫描即使最终成功，cohort 仍可能永久停在 running
或 partial，破坏 component -> task -> coverage/raw 的审计链。

## 2. 改完预期效果如何

- 所有 latest follow-up 继承原 task 的 cohort/component ID。
- follow-up 的 coverage 和 HTTP attempt 继续归属同一 component。
- 原 component 可跨多个时间片累计页数和 raw 计数，并在最终结果后正确聚合父 cohort。
- 旧独立 latest task 的两个字段仍为 NULL，行为保持不变。

## 3. 如何改的/改进思路

回归测试直接创建一个带 cohort/component ID 的 latest task，调用真实
`_enqueue_followup()`，先证明新 task 的两个字段都是 NULL。修复只在 repository enqueue
调用中透传原 task 的两个字段，不改变 latest cursor、frontier、time budget 或
idempotency key 语义。随后运行完整 latest worker 与 cohort worker 回归。
