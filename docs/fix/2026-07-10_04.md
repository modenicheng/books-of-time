# Fix: CAPTCHA 在 HTTP 403 时误判为 FORBIDDEN

## 0. 改的代码大致位置
`books_of_time/http/errors.py` — `_classify_kind()` 函数

## 1. 为什么改
`_classify_kind` 先检查 `status_code == 403` 返回 `FORBIDDEN`，然后才检查响应体中的 captcha 关键词。Bilibili 常以 HTTP 403 + "captcha" / "验证码" / "风控" 返回验证码墙，这被错误分类为 `FORBIDDEN`（永久封禁）而非 `CAPTCHA`（可恢复的验证码挑战）。

后果：
- `RequestBackoffState` 对 `FORBIDDEN` 使用更长的默认 backoff（1800s vs captcha 默认值），使采集不必要的长时间停滞
- 操作日志中错误标记为封禁而非验证码，不利于运维排查

## 2. 改完预期效果如何
HTTP 403 + captcha 文字的响应正确分类为 `CAPTCHA`，使用相应的 backoff 策略（默认 3600s，可独立调整）。

## 3. 如何改的/改进思路
将 body 文字检查移到 status_code 检查之前：
```python
text = body.decode("utf-8", errors="ignore").lower()
if "captcha" in text or "验证码" in text or "风控" in text:
    return RequestErrorKind.CAPTCHA

if status_code == 403:
    return RequestErrorKind.FORBIDDEN
...
```
这样 body 中包含 captcha 标记的响应始终被识别为 `CAPTCHA`，无论 HTTP 状态码如何。纯 HTTP 403（无 captcha 文字）仍正确分类为 `FORBIDDEN`。
