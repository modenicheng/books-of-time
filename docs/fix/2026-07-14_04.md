# Fix: Recovery 与当前 Routine 生成重复采集任务

## 0. 改的代码大致位置

- `books_of_time/task_orchestrator/snapshot_cohort_planner.py`
- `tests/test_snapshot_cohort_planner.py`

## 1. 为什么改

checkpoint 超过允许延迟后，planner 会创建一个当前时间的 recovery cohort，用于
补采所有缺失 component。原实现随后仍继续判断 `routine_due`，并创建同一时间桶的
routine cohort。两个 cohort 的 key 不同，component task 的 idempotency key 也不同，
因此 live 物化会同时生成两套 `video_metrics`、`hot_core` 和 latest 请求。

这违反了 recovery 本身就是“当前时刻补采”的设计语义，也会浪费统一请求预算并让
同一时间点出现重复平台请求。C3 service 虽然只开放 shadow，但底层 live 物化是 C7
迁移的直接基础，因此必须在迁移前修正。

## 2. 改完预期效果如何

- 新 recovery 的 `scheduled_for` 位于当前 planner bucket 时，同桶 routine 被合并。
- recovery `extra.coalesced_routine_bucket` 明确记录本次合并。
- 一个 overdue checkpoint 首次恢复时只创建 recovery 的三个首任务，不再额外创建
  三个 routine 任务。
- 后续 planner 已进入新时间桶时，不会被旧 recovery 永久阻止 routine 规划。

## 3. 如何改的/改进思路

先增加 live 回归用例，复现一次规划产生 3 个 cohort、9 个 component、6 个 task；
测试要求只保留 missed checkpoint 和 recovery，并且只有 3 个唯一 component task。

修复在 recovery 计划确定后比较其 immutable `scheduled_for` 与当前 30 秒 bucket。
只有二者相同时才把 `coalesced_routine` 置为 true。这样同周期 recovery 吸收 routine，
而旧 recovery 在未来 bucket 不会导致 routine 永久饥饿。
